01 What is NexCare — Project Overview

NexCare is a full-stack web application built for telecom companies in India to manage their customer service operations. The application provides a single unified platform for four different types of users — Admin, Manager, Service Representative, and Customer — each with their own dedicated dashboard and strictly controlled access.

Before NexCare, customer service teams would use separate tools: spreadsheets for tickets, email for customer communication, and manual reports for performance. NexCare replaces all of this with one real-time, secure, role-based system.

What Each Role Can Do
RoleWhat they can do
AdminAdd/edit/delete employees, see all tickets, report service outages, view company-wide analytics with charts
ManagerMonitor team ticket performance, view rep metrics, manage outages, see team-level analytics
RepresentativePick up customer tickets, update ticket status, search customers, track their own performance stats
CustomerRaise support tickets, track ticket status, rate resolved tickets, check for outages in their city, use the chatbot

02 How It All Works — The Big Picture

Think of NexCare as two separate programs running at the same time and talking to each other:

Frontend (Angular) runs in the user's browser on port 4200. It shows the UI.
Backend (Spring Boot) runs as a Java server on port 8080. It handles data.
When Angular needs data, it sends an HTTP request to Spring Boot, which reads/writes the database and sends back a JSON response.

The Complete Request Journey
1
User opens browser → Angular loads
The browser loads index.html from the Angular app. Angular's router reads the URL and decides which page component to show. If the URL is /#/login, it loads the Login page.
2
User enters email + password → clicks Sign In
The login form collects credentials and calls AuthService.login() which sends a POST /api/auth/login request to the Spring Boot server (via the proxy on port 8080).
3
Spring Boot validates credentials
AuthController receives the request, calls AuthService, which uses BCryptPasswordEncoder to verify the hashed password stored in the H2 database. If correct, JwtUtil generates a signed token containing the user's email, role, and userId.
4
Token returned to Angular → stored in sessionStorage
The JSON response contains { token: "eyJ...", user: { name, email, role } }. Angular stores both in sessionStorage (wiped when the browser tab closes).
5
Angular redirects to the correct dashboard by role
The login component reads user.role and calls router.navigate(['/admin']) or ['/customer'] etc. The Auth Guard checks the token and role before allowing the page to load.
6
Dashboard loads → fetches data from backend
Every API call automatically gets the JWT token attached to its headers by the JWT Interceptor. Spring Boot's JwtAuthFilter validates this token on every request before allowing any data to be returned.
7
User interacts → Angular updates UI, Spring Boot updates database
Whether raising a ticket, updating a status, or reporting an outage — Angular sends the right API call, Spring Boot processes it, saves to H2, and returns the updated data. Angular re-renders the UI automatically.

SECURITYconfig/SecurityConfig.java
What it does

Configures the entire security layer of the application. It tells Spring Security which URLs are open to everyone (login, register) and which require authentication (everything else). It also registers the JWT filter so every request gets its token validated.

How it works

Public endpoints: POST /api/auth/login and POST /api/auth/register — no token needed.
All other endpoints: Must have a valid JWT token. If not, Spring returns 401 Unauthorized automatically.
CORS: Configured to allow the Angular app on port 4200 to call the backend on port 8080.
Password encoder: Registers BCryptPasswordEncoder as a Spring Bean so it can be injected anywhere in the app.

JWTconfig/JwtUtil.java
What it does

The only class that creates and reads JWT tokens. Think of a JWT token like a signed membership card — JwtUtil is the machine that prints and verifies these cards.

Security Filter
FILTERfilter/JwtAuthFilter.java
What it does

This class runs for every single HTTP request that comes into the backend. It extends OncePerRequestFilter which guarantees it runs exactly once per request. It reads the Authorization: Bearer <token> header, validates the token, and if valid, tells Spring Security who the user is.

Step by step

1. Read the Authorization header from the incoming request.
2. If the header is missing or doesn't start with "Bearer ", skip validation and continue (Spring Security will block it later if the route is protected).
3. Extract the email from the token using JwtUtil.
4. Load the user from the database using that email.
5. Check if the token is valid for that user.
6. If valid, create a UsernamePasswordAuthenticationToken and put it in Spring's SecurityContextHolder — this is how Spring knows the request is authenticated.
7. Continue to the actual controller.

Services (Business Logic)

Services contain the actual business logic. Controllers call services, services call repositories. This separation keeps the code clean and testable.

SERVICEservice/AuthService.java
Handles all authentication operations

login(): Finds user by email, uses BCryptPasswordEncoder to check if the submitted password matches the stored hash. If match, calls JwtUtil.generateToken() and returns LoginResponse with the token and user object.
register(): Checks if email already exists, hashes the password with BCrypt, creates a new CUSTOMER user, saves to database.
changePassword(): Verifies current password, hashes new password, saves.
updateProfile(): Updates name, phone, location for the logged-in user.

SERVICEservice/TicketService.java
Handles all ticket operations

getTicketsForUser(): Uses a switch on the user's role to return the right tickets. Customers see only their own tickets. Reps see tickets assigned to them. Managers see their team's tickets. Admins see all. This method has a @CircuitBreaker annotation — if it fails repeatedly, the fallback returns an empty list instead of crashing.

createTicket(): Builds a new Ticket object from the customer's data and the request form, saves it, then sends a notification to all reps that a new ticket has been raised.

updateStatus(): When a rep sets a ticket to IN_PROGRESS, it records their name, the firstResponseAt time, and calculates responseTimeHrs. When set to CLOSED, it records resolvedAt and calculates resolutionTimeHrs. Notifies the customer of the status change.

rateTicket(): Verifies the customer owns the ticket, then saves the 1–5 star rating.

SERVICEservice/AnalyticsService.java
Calculates all dashboard statistics

getAdminStats(): Returns total counts of customers, managers, reps, tickets by status, average satisfaction rating, and a map of tickets grouped by domain and location for the charts.

getManagerStats(): Returns their team size, team ticket counts, and for each rep: their open/in-progress/closed ticket count plus average response and resolution times.

getRepStats(): Returns this rep's own ticket counts and average response/resolution times.

getCustomerStats(): Returns this customer's own ticket counts by status.

SERVICEservice/UserDetailsServiceImpl.java
Required by Spring Security

Implements Spring's UserDetailsService interface. Spring Security calls loadUserByUsername(email) during JWT validation to load the user from the database. Returns a UserDetails object that Spring uses internally to know what roles the user has.

Security (Frontend)
GUARDapp/guards/auth.guard.ts
Protects routes — runs before any page loads
export const authGuard: CanActivateFn = (route) => {
  const token = sessionStorage.getItem('token');
  const user  = JSON.parse(sessionStorage.getItem('user') || 'null');
  if (!token || !user) { router.navigate(['/login']); return false; }
  const requiredRole = route.data?.['role'];
  if (requiredRole && user.role !== requiredRole) { router.navigate(['/login']); return false; }
  return true;
};

Two checks: (1) Is there a token in sessionStorage? (2) Does the user's role match the required role for this route? Both must pass or the user is sent back to login. This runs on every navigation — not just the first load.

INTERCEPTORapp/interceptors/jwt.interceptor.ts
Automatically adds the JWT token to every HTTP request
export const jwtInterceptor: HttpInterceptorFn = (req, next) => {
  const token = sessionStorage.getItem('token');
  if (token) {
    req = req.clone({ setHeaders: { Authorization: `Bearer ${token}` } });
  }
  return next(req);
};

This is a functional interceptor (Angular 17 style — no class needed). Every time Angular makes an HTTP call, this runs first. If a token exists, it clones the request and adds the Authorization header. The original request is immutable — that's why it uses req.clone().

08 Key Flows — Step by Step
Flow 1: Customer Raises a Support Ticket
1
Customer clicks "My Tickets" tab → clicks "Raise Ticket" button
Angular router navigates to ?tab=tickets. Customer fills in domain (e.g., Internet Services), subject, and description.
2
Angular calls POST /api/tickets
JWT Interceptor adds the token. Spring Boot receives the request. JwtAuthFilter validates the token and confirms the caller is a CUSTOMER.
3
TicketController → TicketService.createTicket()
Builds a Ticket object with status=OPEN, customerId from the token, and all form fields. Saves to database.
4
Notifications sent to all reps
Service fetches all REPRESENTATIVE users and creates a notification for each: "New ticket #7 raised by Priya Sharma — Internet Services".
5
Response returned to Angular
The saved ticket is returned as JSON. Angular adds it to the tickets list. A toast notification shows "Ticket raised successfully".
Flow 2: Rep Updates Ticket Status to IN_PROGRESS
1
Rep clicks on an OPEN ticket → selects "In Progress" → submits
Angular calls PUT /api/tickets/{id}/status with body { "status": "IN_PROGRESS" }
2
TicketService.updateStatus() runs
Checks that this is the first response (firstResponseAt == null). Records: rep's name and ID, current timestamp as firstResponseAt. Calculates responseTimeHrs = (now - createdAt) in hours.
3
Notification sent to customer
"Your ticket #3 status updated to IN PROGRESS" — saved to NOTIFICATIONS table.
4
Customer sees update
Next time the customer loads their tickets (or refreshes), the ticket shows "In Progress". The bell icon in their topbar shows a new notification count.
Flow 3: Admin Reports a Service Outage
1
Admin fills the outage form: city=Mumbai, domain=Internet Services, severity=HIGH
Angular calls POST /api/outages with the form data.
2
OutageController → OutageService saves the outage
Status is set to ACTIVE. reportedAt is recorded.
3
Mumbai customers see an alert banner on next dashboard load
Customer dashboard calls GET /api/outages/location?city=Mumbai. Returns the active outage. Angular displays: "Service Disruption — Internet Services disruption in Andheri, Bandra, Colaba. Our team is working to restore service."
4
Admin resolves the outage
PUT /api/outages/{id}/resolve. Status becomes RESOLVED, resolvedAt recorded. Customers in Mumbai no longer see the banner.
09 Security Architecture
Why JWT (JSON Web Tokens)?

Traditional session-based auth stores login state on the server. With JWT, the server is completely stateless — all user information is encoded inside the token itself. The server just needs to verify the signature. This scales better and fits REST API design perfectly.

What's inside a JWT token?

A JWT has three parts separated by dots: header.payload.signature

// Decoded payload of a NexCare JWT token:
{
  "sub": "amit.kumar@nexcare.in",   ← user's email (subject)
  "userId": 5,                       ← user's database ID
  "role": "REPRESENTATIVE",          ← their role
  "name": "Amit Kumar",              ← display name
  "managerId": 3,                    ← which manager they report to
  "iat": 1718000000,                 ← issued at (Unix timestamp)
  "exp": 1718086400                  ← expires after 24 hours
}

The token is signed with a secret key using HMAC-SHA256. Any tampering with the payload would invalidate the signature — the server detects this immediately and returns 401.

Dual Security — Frontend and Backend
LayerWhereWhat it doesCan be bypassed?
Auth GuardAngular (frontend)Prevents loading protected pages without a valid sessionYes — by a technical user. That's why the backend layer exists.
JWT FilterSpring Boot (backend)Validates token signature on every API call. No valid token = 401 Unauthorized.No — the backend is the final authority.
@PreAuthorizeSpring Boot (per-method)Checks role even after JWT validation. Blocks cross-role API calls.No.

Why two layers? The auth guard is for user experience — it prevents the UI from even trying to show an admin page to a customer. The JWT filter is for real security — even if someone uses Postman or a custom script to call the API, they cannot get any data without a valid token for the correct role.

10 Configuration Files
CONFIGfrontend/proxy.conf.json
{
  "/api": {
    "target": "http://localhost:8080",
    "secure": false,
    "changeOrigin": true
  }
}

Angular's development server runs on port 4200. Spring Boot runs on port 8080. Browsers block requests from one port to another (CORS policy). This proxy configuration solves that — any request from Angular that starts with /api is silently forwarded to http://localhost:8080. To the browser, everything looks like it's on port 4200.

CONFIGfrontend/angular.json

The main Angular build configuration. Key settings: proxyConfig: "proxy.conf.json" registers the proxy, outputPath: "dist/frontend" sets where the built files go, budgets warns if the bundle gets too large. The styles array points to src/styles.css as the global stylesheet.

CONFIGfrontend/package.json

Lists all JavaScript/TypeScript packages the frontend needs. Key dependencies: @angular/core, @angular/router, @angular/forms, @angular/common/http (all @17.x), chart.js for the charts, rxjs for reactive programming. Running npm install reads this file and downloads everything listed.

CONFIGpom.xml (backend)

Maven's equivalent of package.json. Lists all Java dependencies: spring-boot-starter-web (REST API), spring-boot-starter-security (security), spring-boot-starter-data-jpa (database), h2 (H2 database driver), jjwt-api/impl/jackson 0.12.3 (JWT library), lombok (code generation), resilience4j-spring-boot3 (circuit breaker). Maven downloads all of these from Maven Central on first run.

CONFIG.vscode/tasks.json

Defines run tasks for VS Code. The main task "Run NexCare (Full Stack)" is a compound task that starts both "Backend: Spring Boot" (mvn spring-boot:run) and "Frontend: Angular" (npm install && npx ng serve --port 4200 --open) simultaneously. These can be triggered from the VS Code menu: Terminal → Run Task.
